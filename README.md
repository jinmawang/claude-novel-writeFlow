# 📖 claude-novel-writeFlow

**一套基于 Claude Code 的 AI 小说写作工作流插件**

通过三个自定义斜杠命令，将 AI 辅助写作提升到新高度——从大纲构思到章节写作，每一步都有专属的多智能体系统为你把关。

---

## ✨ 核心优势

### 🧠 多智能体头脑风暴（Brainstorm）
大纲自动生成模式会并行启动 **4 个专属 AI 智能体**，分别从市场调研、人物弧线、叙事结构、主题立意四个维度独立发散，最终合成 5 种差异化方案供你选择，告别千篇一律的故事框架。

### 🔍 三智能体写作审核架构
每章写作都会经历三重智能体协作：
- **Writer Agent**：专注创作，全力输出
- **Style Reviewer**：专门检测 AI 机械感，确保文字有"人味"，严格贴合你的风格规范
- **Continuity Reviewer**：对照前后 2 章大纲，确保情节衔接自然，伏笔呼应到位

三者形成**闭环修改机制**——审核不通过，自动反馈修改，最多 3 轮迭代，直到达标才保存。

### 📂 结构化大纲管理
大纲以**文件夹**形式存储，整体大纲与每章细纲分离管理，写作时精准读取上下文（前后各 2 章大纲），做到局部创作与全局把控两不误。

### ⚡ 全自动写作模式
所有命令均支持 `--auto` 参数，从大纲生成到全书写作，可实现高度自动化，大幅降低人工干预成本。

## 🚀 快速开始

### 安装 Claude Code

如果还没有安装 Claude Code，先完成以下步骤：

```bash
# 安装 Claude Code CLI
npm install -g @anthropic-ai/claude-code

# 登录（首次使用需要）
claude
```

或者直接下载 [Claude Code 桌面版](https://claude.ai/code)，安装后登录即可。

---

### 安装本插件

**插件原理：** Claude Code 会自动识别项目目录下 `.claude/commands/` 文件夹中的 Markdown 文件，将其注册为可用的斜杠命令。安装本插件就是把这个文件夹放进你的小说项目目录。

#### 方案一：新建小说项目（推荐）

直接把本仓库作为你的小说项目目录使用：

```bash
# 克隆到你的小说目录（my-novel 改成你想要的名字）
git clone https://github.com/YOUR_USERNAME/claude-novel-writeFlow.git my-novel
cd my-novel

# 用 Claude Code 打开
claude .
```

打开后输入 `/` 即可看到 `/outline`、`/style`、`/write` 等命令出现在补全列表中，说明安装成功。

#### 方案二：为已有项目添加插件

如果你已经有一个小说项目目录，只需复制命令文件夹：

```bash
# 进入你已有的小说项目目录
cd /path/to/your-novel

# 复制命令文件夹
git clone https://github.com/YOUR_USERNAME/claude-novel-writeFlow.git /tmp/writeflow
cp -r /tmp/writeflow/.claude .
rm -rf /tmp/writeflow

# 用 Claude Code 打开
claude .
```

#### 验证安装

在 Claude Code 中输入 `/`，补全列表中出现以下命令即为安装成功：

```
/outline   /style   /write   /status   /export
```

### 推荐工作流

```
第一步：/style          → 设定写作风格
第二步：/outline        → 创建小说大纲
第三步：/write --auto   → 自动写作全书
第四步：/status         → 查看写作进度
第五步：/export         → 导出完整稿件
```

## 📋 命令详解

### `/outline` — 大纲创作

```
/outline [--auto] [初始描述]
```

```
/outline --chapter=N
```

> 仅生成或重写第 N 章的细纲，不影响整体大纲和其他章节。

**交互模式**（默认）：通过对话逐步完善你的故事框架，随时修改，满意后一键生成大纲文件夹。

**自动模式** `--auto`：

1. **5 轮定向问答**：收集故事类型、核心冲突、人物设定、篇幅规划、特殊要求
2. **4 路并行 Brainstorm**：
   - 🔍 **市场调研 Agent**：搜索分析同类型近期成功作品，提炼读者偏好和平台规律
   - 👤 **人物弧线 Agent**：发散 5+ 种主角成长路径和人物关系方案
   - 🏗️ **叙事结构 Agent**：探索线性/多线/非线性等 5+ 种叙事架构
   - 💡 **主题立意 Agent**：挖掘故事可能承载的不同深度与情感内核
3. **收敛生成**：汇总 4 路结果，合成 5 个差异化完整方案
4. **用户选择**：选择、融合或修改，直到满意
5. **自动保存**：生成完整大纲文件夹

**生成文件：**
```
outline/
├── overview.md         # 整体大纲（人物档案、三幕结构、章节总览）
├── chapter-01.md       # 第1章细纲（含前后章衔接点）
├── chapter-02.md
└── ...
```

---

### `/style` — 写作风格规范

```
/style [风格描述或参考作者]
```

通过 5 轮问答定义你专属的写作风格，支持参考任意作家/作品（自动联网分析其写法特征）。

生成的 `style-rules.md` 涵盖 **10 个维度**：

| 维度 | 内容 |
|------|------|
| 叙事视角与声音 | 视角类型、叙述距离、信息管控策略 |
| 句式节奏 | 长短句比例、段落控制、标点节奏 |
| 对话风格 | 标签用法、潜台词处理、人物个性化口吻 |
| 场景描写 | 细节选择原则、感官优先级、氛围营造 |
| 情感表达 | 直接/间接比例、高情绪时刻处理 |
| 词汇与语调 | 文白混用规则、禁用词汇清单 |
| 中文特有技法 | 留白、意象体系、叙述落差 |
| 章节写作规范 | 开头/结尾原则、节奏分配 |
| 正反对比示例 | 3 组以上"推荐 vs 避免"具体示例 |
| AI味检测清单 | 8 条机械感排查项，写后自查用 |

### `/write` — 章节写作

```
/write [章节号] [--words=3000] [--auto]
```

| 参数 | 说明 |
|------|------|
| `章节号` | 指定写第几章，不填则自动写下一章 |
| `--words=N` | 目标字数（默认 3000，允许 ±500 浮动） |
| `--auto` | 自动连续写作，直到全书完成 |
| `--no-review` | 跳过审核智能体，直接保存初稿（适合粗稿阶段） |
| `--auto --force` | 无干预连续写作，章与章之间不暂停确认 |

**三智能体写作架构：**

```
主协调者
  │
  ├─► Writer Agent
  │       输入：整体大纲 + 本章大纲 + 前后±2章大纲 + 近2章正文 + 风格规范
  │       输出：初稿
  │
  ├─► Style Reviewer（并行）
  │       检查 AI 机械感（8项）+ 风格规范符合度（4项）
  │       → 问题列表 + 具体修改建议
  │
  └─► Continuity Reviewer（并行）
          检查大纲符合度（4项）+ 前后章衔接（5项）+ 整体一致性（2项）
          → 问题列表 + 具体修改建议

  若有问题 → 反馈 Writer Agent 修改 → 再次审核（最多 3 轮）
  若通过   → 保存到 chapters/chapter-XX.md
```

**自动模式**下，写完每章后自动分析下一章，循环写作直到全书完成，每章之间有确认节点可随时介入。

### `/context` — 上下文管理

```
/context [--init] [--rebuild]
```

| 参数 | 说明 |
|------|------|
| 无参数 | 自动模式：context 不存在则初始化，否则从已有章节重建 |
| `--init` | 仅创建空的 context 文件，不分析章节 |
| `--rebuild` | 清空并从全部已有章节重新提取，适合中途引入本功能时使用 |

**日常使用无需手动运行**——每次 `/write` 完成后自动提取当章新增事实。

以下情况需要手动运行：
- 项目已有多章内容，首次引入 context 功能时：运行 `/context` 初始化
- 发现 context 记录有误需要重建：运行 `/context --rebuild`

---

### `/status` — 写作进度

```
/status
```

扫描项目目录，输出结构化进度报告：

- 整体大纲和风格规范的创建状态
- 章节大纲规划数量 vs 预计总章节数
- 已写章节数量和总字数
- 待写章节列表（有大纲但尚未写作的）
- 缺少大纲的章节提示

---

### `/export` — 导出完整稿件

```
/export [--output=文件名] [--format=markdown|txt]
```

| 参数 | 说明 |
|------|------|
| `--output=文件名` | 导出文件名（不含扩展名），默认 `manuscript` |
| `--format=markdown\|txt` | 导出格式，默认 `markdown` |

将所有已写章节按编号顺序合并为单一文件，输出到项目根目录。章节之间以分隔线隔开。

## 💡 使用技巧

**关于大纲细化程度**
`/write` 写作时会同时读取当前章节及前后各 2 章的细纲。大纲写得越详细（尤其是"前后章衔接点"部分），写出来的章节过渡就越自然流畅。

**关于风格规范的 AI 味检测**
`style-rules.md` 末尾的"AI味检测清单"会被 Style Reviewer 严格执行。如果你有特别敏感的 AI 感知点，可以手动编辑 `style-rules.md` 在清单中添加。

**关于字数控制**
允许 ±500 字浮动，意味着 3000 字目标实际范围是 2500-3500 字。如果大纲该章节内容特别丰富，建议设置 `--words=4000` 避免压缩。

**关于自动模式**
`/write --auto` 每章完成后都会暂停确认，你可以随时叫停、阅读已完成章节后再继续，不会失控狂奔。

**关于上下文追踪**
`context/` 文件夹由 `/write` 自动维护，记录正文中已确立的人物细节、世界设定、关键事件和时间线。Continuity Reviewer 会在每章审核时比对这些记录，防止前后矛盾。如果中途引入该功能（已有多章），记得先运行 `/context` 初始化存量章节的上下文。
